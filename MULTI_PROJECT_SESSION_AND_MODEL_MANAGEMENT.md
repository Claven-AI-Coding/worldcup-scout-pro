## ❓ 问题 1：多项目切换时，上一个未结束的会话会结束吗？

### 答案：**不会自动结束，但需要明确管理**

## 📋 会话的生命周期

### 会话的三种状态

```
1. 活跃状态（Active）
   - 会话正在进行中
   - 可以继续对话
   - 占用内存和 token 预算

2. 暂停状态（Paused）
   - 会话被暂停
   - 不占用 token
   - 可以随时恢复

3. 结束状态（Closed）
   - 会话已结束
   - 不占用资源
   - 只能查看历史
```

### 多项目场景下的会话管理

```
场景：同时进行 3 个项目

项目 A（世界杯球探 Pro）
  ├─ 会话 A1（进行中）
  ├─ 会话 A2（已结束）
  └─ 会话 A3（已结束）

项目 B（产品 B）
  ├─ 会话 B1（进行中）
  └─ 会话 B2（已结束）

项目 C（产品 C）
  └─ 会话 C1（进行中）

当前活跃会话：A1、B1、C1（3 个）
```

## 🔄 会话切换的完整流程

### 流程 1：项目切换（从项目 A 到项目 B）

```
步骤 1：评估项目 A 的状态
  - 当前任务是否完成？
  - 是否需要保存状态？
  - 是否需要压缩历史？

步骤 2：决定项目 A 会话的处理方式
  
  选项 A：暂停会话（推荐）
    /stop（但不完全关闭）
    → 会话进入暂停状态
    → 可以随时恢复
    → 不占用 token
  
  选项 B：完全结束会话
    /stop
    → 会话完全结束
    → 只能查看历史
    → 不能恢复
  
  选项 C：继续保持活跃
    （不执行任何命令）
    → 会话继续活跃
    → 占用 token 预算
    → 可以随时切回

步骤 3：切换到项目 B
  /new
  → 创建新的会话
  → 项目 B 的会话开始
  → 项目 A 的会话状态保持不变

步骤 4：通知 GraySon
  🔄 会话切换通知
  从：项目 A（会话 A1）
  到：项目 B（会话 B1）
  项目 A 状态：[暂停/结束/继续活跃]
```

## 📊 多项目会话管理的最佳实践

### 推荐的会话管理策略

**策略 1：暂停-恢复模式（推荐）**

```
项目 A 进行中
  ↓
需要切换到项目 B
  ↓
执行 /stop（暂停项目 A）
  ↓
执行 /new（开始项目 B）
  ↓
项目 B 进行中
  ↓
需要切回项目 A
  ↓
恢复项目 A 的会话
  ↓
继续项目 A 的工作

优点：
  - 可以随时切换
  - 保留所有历史
  - 节省 token
  - 灵活高效

缺点：
  - 需要手动管理
  - 需要记录会话 ID
```

**策略 2：完全隔离模式**

```
项目 A 进行中
  ↓
执行 /stop（完全结束项目 A）
  ↓
执行 /new（开始项目 B）
  ↓
项目 B 进行中
  ↓
需要切回项目 A
  ↓
执行 /new（创建新的项目 A 会话）
  ↓
重新加载项目 A 的上下文
  ↓
继续项目 A 的工作

优点：
  - 完全隔离
  - 不会相互干扰
  - 清晰明确

缺点：
  - 需要重新加载上下文
  - 浪费 token
  - 效率较低
```

**策略 3：并行活跃模式**

```
项目 A 会话保持活跃
  ↓
执行 /new（创建项目 B 会话）
  ↓
项目 B 会话活跃
  ↓
项目 A 和 B 同时活跃
  ↓
可以在两个会话间快速切换
  ↓
完成后分别 /stop

优点：
  - 快速切换
  - 不需要重新加载
  - 适合频繁切换

缺点：
  - 占用更多 token
  - 成本更高
  - 容易混淆
```

## 🎯 Claven 的会话管理建议

### 根据项目类型选择策略

**同步项目（紧密协作）**
```
例如：世界杯球探 Pro

使用：暂停-恢复模式
原因：
  - 需要频繁切回
  - 需要保留完整历史
  - 需要节省 token

流程：
  1. 项目 A 进行中
  2. 需要处理紧急事务
  3. /stop（暂停项目 A）
  4. /new（处理紧急事务）
  5. 完成后恢复项目 A
```

**异步项目（相对独立）**
```
例如：产品 B、产品 C

使用：完全隔离模式
原因：
  - 相对独立
  - 不需要频繁切换
  - 清晰明确

流程：
  1. 项目 A 进行中
  2. 完成项目 A 的阶段
  3. /stop（完全结束项目 A）
  4. /new（开始项目 B）
  5. 项目 B 完成后再切回
```

**高频切换场景**
```
例如：多个紧急任务

使用：并行活跃模式
原因：
  - 需要频繁切换
  - 需要快速响应
  - 时间紧张

流程：
  1. 项目 A 会话保持活跃
  2. /new（创建项目 B 会话）
  3. 在 A、B 间快速切换
  4. 完成后分别 /stop
```

## ❓ 问题 2：不同 Agent 的模型不一样，后一个 Agent 的模型会覆盖前一个吗？

### 答案：**不会覆盖，每个 Agent 有独立的模型配置**

---

## 🤖 Agent 模型配置的隔离机制

### 模型配置的作用域

```
全局配置（会话级别）
  /model opus
  ↓
  影响范围：当前会话中的所有操作
  ↓
  如果没有特殊指定，所有 Agent 都使用这个模型

Agent 级别配置（Agent 级别）
  CodeSmith 使用 opus
  UIArtisan 使用 sonnet
  TestMaster 使用 haiku
  ↓
  影响范围：只影响该 Agent
  ↓
  覆盖全局配置
```

### 模型配置的优先级

```
优先级从高到低：

1. Agent 专用模型（最高）
   - 在任务分配单中指定
   - 例如：CodeSmith 使用 opus
   - 覆盖所有其他配置

2. 任务级别模型（中等）
   - 在任务中指定
   - 例如：这个任务用 sonnet
   - 覆盖全局配置

3. 全局模型配置（最低）
   - /model opus
   - 影响所有 Agent
   - 如果没有特殊指定就用这个
```

## ❓ 问题 3：Agent 的模型相同的情况下，是相同 Agent 都在工作吗？

### 答案：**不是，每个 Agent 是独立的个体**

## 🤖 Agent 的独立性和并行工作

### Agent 的本质

```
每个 Agent 是一个独立的工作单元：

CodeSmith（后端开发）
  - 独立的身份
  - 独立的能力
  - 独立的工作
  - 独立的输出

UIArtisan（前端开发）
  - 独立的身份
  - 独立的能力
  - 独立的工作
  - 独立的输出

即使他们使用相同的模型，他们仍然是不同的 Agent
```

### 模型相同 ≠ Agent 相同

```
场景：CodeSmith 和 UIArtisan 都使用 sonnet

CodeSmith 使用 sonnet
  - 用于后端开发
  - 处理 API 逻辑
  - 输出后端代码

UIArtisan 使用 sonnet
  - 用于前端开发
  - 处理 UI 逻辑
  - 输出前端代码

虽然都用 sonnet，但：
  - 他们是不同的 Agent
  - 他们做不同的工作
  - 他们输出不同的结果
  - 他们互不影响
```

### 多 Agent 并行工作的真实情况

```
时间：09:00-12:00

CodeSmith 在做什么：
  - 设计 API 接口
  - 实现认证逻辑
  - 编写数据库查询
  - 使用 opus 模型

UIArtisan 在做什么：
  - 设计登录页面
  - 实现表单验证
  - 优化 UI 样式
  - 使用 sonnet 模型

TestMaster 在做什么：
  - 编写测试用例
  - 设计测试场景
  - 生成测试报告
  - 使用 haiku 模型

结果：
  - 3 个 Agent 同时工作
  - 3 个不同的任务
  - 3 个不同的输出
  - 3 个不同的模型
  - 完全独立
```

### 任务分配时的模型选择

```
原则：
  1. 根据任务复杂度选择模型
  2. 不要所有 Agent 都用同一个模型
  3. 充分利用不同模型的优势
  4. 平衡成本和质量

建议分配：

  - 复杂度：高
  - 推荐模型：opus
  - 原因：核心功能需要最好的质量

前端开发（UIArtisan）

  - 推荐模型：sonnet
  - 原因：平衡成本和质量

测试编写（TestMaster）
  - 复杂度：低
  - 推荐模型：haiku

文档生成（DocWriter）
  - 复杂度：低
  - 推荐模型：haiku
```
