## 🚀 快速开始

### 最常用的 5 个命令

```
1. /status
   查看当前状态（token、时间、成本）
   用途：了解资源使用情况
   频率：每小时检查一次

2. /model <id>
   切换 AI 模型
   用途：根据任务选择合适的模型
   频率：按需切换

3. /think <level>
   设置推理深度
   用途：控制 AI 的思考时间
   频率：按任务复杂度调整

4. /new
   创建新会话
   用途：开始全新的项目或任务
   频率：项目切换时使用

5. /verbose on|off
   控制输出详细程度
   用途：减少或增加输出信息
   频率：按需调整
```

## 📋 四大命令类别

### 1️⃣ 会话管理（Session）

**用途：** 管理对话会话的生命周期

| 命令 | 中文 | 说明 | 何时使用 |
|------|------|------|---------|
| `/new` | 新建会话 | 创建全新的会话，清空所有对话历史 | 开始新项目、切换产品 |
| `/reset` | 重置会话 | 清空当前会话的上下文，但保留会话 | 切换任务、清理环境 |
| `/compact [instructions]` | 压缩历史 | 压缩会话历史以节省 token | token 接近上限时 |
| `/stop` | 停止会话 | 结束当前会话，保存状态 | 完成工作、下班前 |

**详细说明：**

```
/new
  - 创建完全新的会话
  - 清空所有对话历史
  - 不清空文件系统、Git、内存
  - 新会话会自动读取 MEMORY.md
  - 适合：开始新项目、完全切换任务

/reset
  - 清空当前会话的上下文
  - 保留会话本身
  - 适合：在同一会话中切换任务

/compact [instructions]
  - 压缩会话历史
  - 节省 token 使用
  - 可选参数：指定保留的内容
  - 例如：/compact "保留最近的代码和决策"
  - 适合：token 使用接近上限

/stop
  - 结束当前会话
  - 保存会话状态
  - 适合：完成工作、下班前
```

### 2️⃣ 选项配置（Options）

**用途：** 配置 AI 的工作参数

| 命令 | 中文 | 说明 | 可选值 |
|------|------|------|--------|
| `/think <level>` | 思考深度 | 设置 AI 的推理深度 | off / fast / deep / extended |
| `/model <id>` | 模型选择 | 切换使用的 AI 模型 | opus-thinking / opus / sonnet / haiku |
| `/verbose on\|off` | 详细模式 | 控制输出的详细程度 | on / off |

**详细说明：**

```
/think <level>
  
  off
    - 不使用推理
    - 最快速度
    - 最低成本
    - 适合：简单任务、文档生成
    - 例如：/think off
  
  fast
    - 快速推理
    - 平衡速度和质量
    - 中等成本
    - 适合：一般开发任务
    - 例如：/think fast
  
  deep
    - 深度推理
    - 较慢速度
    - 较高成本
    - 适合：复杂问题、关键决策
    - 例如：/think deep
  
  extended
    - 扩展推理
    - 最慢速度
    - 最高成本
    - 适合：超复杂问题、架构设计
    - 例如：/think extended

---

/model <id>

  opus-thinking
    - 最强推理模型
    - 最慢速度
    - 最高成本
    - 适合：架构设计、安全分析
    - 例如：/model opus-thinking
  
  opus
    - 最强通用模型
    - 中等速度
    - 高成本
    - 适合：核心功能开发、代码审查
    - 例如：/model opus
  
  sonnet
    - 平衡模型
    - 快速速度
    - 中等成本
    - 适合：一般开发、API 设计
    - 例如：/model sonnet
  
  haiku
    - 最便宜模型
    - 最快速度
    - 最低成本
    - 适合：简单任务、文档、协调
    - 例如：/model haiku

---

/verbose on|off

  on
    - 显示所有细节
    - 显示工具调用
    - 显示推理过程
    - 显示中间步骤
    - 适合：开发和调试
    - 例如：/verbose on
  
  off
    - 只显示最终结果
    - 隐藏中间过程
    - 减少输出
    - 适合：生产环境
    - 例如：/verbose off
```

### 3️⃣ 状态查询（Status）

**用途：** 查询当前系统状态

| 命令 | 中文 | 说明 | 返回信息 |
|------|------|------|---------|
| `/status` | 查看状态 | 查看会话的详细状态 | token、时间、成本、模型 |
| `/whoami` | 查看身份 | 查看当前用户身份 | 用户名、ID、权限、角色 |
| `/context` | 查看上下文 | 查看当前加载的上下文 | 文件、内存、工作目录、工具 |

**详细说明：**

```
/status
  返回信息：
    - Token 使用情况（已用/总数）
    - 会话时间（开始时间、运行时长）
    - 成本统计（当前成本、预估总成本）
    - 模型信息（当前模型、思考深度）
    - 详细模式（on/off）
  
  用途：
    - 了解资源使用情况
    - 评估是否需要优化
    - 决定是否继续或压缩
  
  使用频率：
    - 每小时检查一次
    - 或在重要决策前检查
  
  例如：/status

---

/whoami
  返回信息：
    - 用户名（GraySon）
    - 用户 ID（5851189964）
    - 权限级别（CEO/Leader/Agent）
    - 角色（需求方/决策者/执行者）
  
  用途：
    - 确认当前身份
    - 验证权限
    - 了解角色
  
  使用场景：
    - 多用户环境中确认身份
    - 权限相关操作前验证
  
  例如：/whoami

---

/context
  返回信息：
    - 加载的文件（SOUL.md、USER.md、MEMORY.md）
    - 内存内容（当前记忆）
    - 工作目录（/root/projects/worldcup-scout-pro）
    - 可用工具（read、write、exec 等）
  
  用途：
    - 了解当前上下文
    - 检查加载的文件
    - 验证工作环境
  
  使用场景：
    - 新会话开始时检查
    - 上下文不清楚时查询
  
  例如：/context
```

## ❓ 常见问题

### Q1：/new 和 /reset 有什么区别？

```
/new
  - 创建全新的会话
  - 清空所有对话历史
  - 适合：完全切换项目

/reset
  - 在同一会话中重置
  - 清空上下文但保留会话
  - 适合：在同一会话中切换任务

选择建议：
  - 不同项目 → /new
  - 同一项目不同任务 → /reset
```

### Q2：/compact 会丢失信息吗？

```
/compact 的行为：
  - 压缩会话历史
  - 保留关键信息
  - 不会丢失已保存的文件
  - 不会丢失 Git 提交
  - 不会丢失内存文件

安全性：
  - 完全安全
  - 可以放心使用
  - 所有重要信息都被保存
```

### Q3：如何选择合适的模型？

```
选择标准：

简单任务（文档、协调）
  → /model haiku

一般开发（API、组件）
  → /model sonnet

核心功能（后端、前端）
  → /model opus

关键决策（架构、安全）
  → /model opus-thinking

不确定时：
  → 先用 sonnet
  → 如果不够好，升级到 opus
  → 如果太贵，降级到 haiku
```

### Q4：什么时候应该压缩历史？

```
压缩时机：

1. Token 接近上限
   - /status 显示 token 使用 >80%
   - 执行 /compact

2. 会话很长
   - 对话超过 100 条
   - 执行 /compact

3. 切换大任务
   - 从一个大项目切换到另一个
   - 执行 /compact

4. 定期维护
   - 每周压缩一次
   - 保持会话健康

压缩命令：
  /compact "保留最近的代码和决策"
```

### Q5：/verbose on 会增加成本吗？

```
/verbose on 的影响：

成本：
  - 不直接增加成本
  - 但会增加输出 token
  - 可能间接增加成本

速度：
  - 输出更多信息
  - 响应可能稍慢

建议：
  - 开发阶段：/verbose on
  - 生产阶段：/verbose off
  - 调试阶段：/verbose on
```

## ✅ 最佳实践

### 1. 定期检查状态

```
建议频率：每小时一次

命令：
  /status

检查内容：
  - Token 使用情况
  - 成本统计
  - 当前模型
  - 思考深度

决策：
  - 如果 token >80% → /compact
  - 如果成本过高 → 切换模型
  - 如果速度不够 → 调整思考深度
```

### 2. 合理选择模型

```
原则：
  - 优先使用便宜模型
  - 只在必要时升级
  - 定期评估是否需要升级

流程：
  1. 用 haiku 尝试
  2. 如果不够好，升级到 sonnet
  3. 如果还不够，升级到 opus
  4. 只在关键决策时用 opus-thinking
```

### 3. 控制输出详细程度

```
原则：
  - 开发时详细
  - 生产时简洁

建议：
  - 开发阶段：/verbose on
  - 测试阶段：/verbose on
  - 生产阶段：/verbose off
  - 调试阶段：/verbose on
```

### 4. 及时压缩历史

```
原则：
  - 定期压缩
  - 保留关键信息
  - 不丢失重要数据

建议：
  - 每周压缩一次
  - Token 接近上限时压缩
  - 切换大任务时压缩

命令：
  /compact "保留最近的代码和决策"
```

### 5. 定期重置和新建

```
原则：
  - 项目切换时新建
  - 任务切换时重置
  - 定期清理环境

建议：
  - 不同项目 → /new
  - 同项目不同任务 → /reset
  - 每周新建一次
  - 每天重置 2-3 次
```
